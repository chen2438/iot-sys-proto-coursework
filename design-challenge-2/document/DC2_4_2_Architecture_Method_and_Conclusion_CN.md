# Design Challenge 2 - Task 4.2 说明文档

本文档总结 Task 4.2 的模型架构设计、训练方法、测试方法与结论，供报告撰写时直接参考。

## 1. 任务目标

Task 4.2 的目标是在提供的 Magic Wand 工作流基础上，对 CNN 架构进行探索，并分析以下三项之间的 trade-off：

- 分类准确率
- 量化后模型大小
- MCU 端推理延迟

本部分不以 baseline 对比为主，而是聚焦于自定义 CNN 架构之间的差异。

## 2. 最终选用的 5 个模型架构

最终报告选用了 5 个量化后准确率落在 70%-90% 区间的模型，这些模型既满足题目要求，也能形成较清晰的性能变化趋势。

| 模型名 | 卷积层配置 | Kernel Size | Dropout | Epochs | 量化测试准确率 (%) | 模型大小 (bytes) | 平均 MCU 延迟 (us) |
|---|---|---:|---:|---:|---:|---:|---:|
| tiny_2 | 8-16 | 3 | 0.25 | 30 | 74.09 | 5536 | 41006.36 |
| mid_12_24 | 12-24 | 3 | 0.28 | 24 | 81.00 | 7544 | 61851.45 |
| mid_16_24 | 16-24 | 3 | 0.30 | 24 | 73.00 | 8624 | 75026.65 |
| mid_14_28 | 14-28 | 3 | 0.31 | 20 | 71.64 | 8744 | 73696.20 |
| mid_14_24_24 | 14-24-24 | 3 | 0.35 | 16 | 87.18 | 14608 | 69963.86 |

这些模型的共同特点是：

- 输入尺寸保持为 `32 x 32 x 3`
- 均为 CNN-based neural network
- 都被量化为 TensorFlow Lite 模型后，再部署到 Arduino Nano 33 BLE Sense 上

## 3. 模型架构设计思路

本次架构探索主要围绕以下几个变量展开：

- 卷积层数量
- 每层卷积通道数
- dropout 强度
- 训练轮数

设计思路如下：

- `tiny_2` 代表较小的轻量模型，用来观察最小资源占用下的可用准确率
- `mid_12_24`、`mid_16_24`、`mid_14_28` 代表中等复杂度模型，用于观察通道数变化对精度和延迟的影响
- `mid_14_24_24` 在保持中等模型规模的同时增加了一层卷积层，用来观察更深结构是否能在可接受代价内提高准确率

整体上，模型容量越大，通常能获得更高的准确率，但也会带来更高的模型大小与更长的推理时间。

## 4. 训练方法

训练过程在 Google Colab 中完成，整体流程保持与提供的 Magic Wand notebook 一致，只在 CNN 架构上进行调整。

### 4.1 数据与预处理

- 使用提供的 Magic Wand digit dataset
- 数据划分比例保持与提供 notebook 一致
- 使用相同的 stroke rasterization 流程，将手势轨迹转换为图像
- 输入图像大小为 `32 x 32`

### 4.2 模型训练设置

所有模型均采用以下训练方法：

- 优化器：`Adam(1e-3)`
- 损失函数：`binary_crossentropy`
- 评估指标：`accuracy`
- 训练框架：TensorFlow / Keras

同时使用了以下训练策略：

- `ModelCheckpoint` 保存验证集表现最好的模型
- `EarlyStopping` 根据 `val_accuracy` 提前停止训练，并恢复最佳权重

这样做的目的是：

- 避免只看最后一个 epoch 导致结果不稳定
- 减少过拟合风险
- 使不同架构之间的比较更稳定

### 4.3 量化与导出

训练完成后，每个模型都经历以下步骤：

1. 保存为 Keras 模型
2. 转换为 float TFLite 模型
3. 进行 integer quantization，生成 quantized TFLite 模型
4. 导出为 `.cc` 文件，供 TensorFlow Lite Micro 在 Arduino 工程中加载

因此，最终部署形式符合题目要求：

`CNN based neural network deployed using TensorFlow Lite Micro`

## 5. 测试方法

本次测试分为 PC 端测试和 MCU 端测试两部分。

### 5.1 PC 端准确率测试

PC 端测试准确率使用 Colab 中的 test dataset 进行评估。

测试方法如下：

- 对 test split 中的全部样本进行评估
- 不使用 confidence threshold
- 不丢弃低分样本
- 所有测试样本都计入 accuracy

这意味着这里的 accuracy 是标准测试准确率，适合用于模型之间的公平比较。

### 5.2 MCU 端延迟测试

MCU 端延迟测试在 Arduino Nano 33 BLE Sense 上进行。

测试流程如下：

1. 将量化模型导出的 `.cc` 文件复制到 `magic_wand_model_data.cpp`
2. 在 `magic_wand.ino` 中围绕 `interpreter->Invoke()` 加入 `micros()` 计时代码
3. 将工程上传到开发板
4. 通过串口监视器记录每次推理输出的 `Latency: xxxx us`
5. 统计每个模型的平均值、最小值和最大值

本次 4.2 的主分析使用平均 MCU latency。

## 6. 结果分析

### 6.1 Accuracy vs Model Size

从结果可以看出，较大的模型通常具有更高的准确率。

- `tiny_2` 模型最小，仅 `5536 bytes`，量化准确率为 `74.09%`
- `mid_12_24` 在模型大小增加到 `7544 bytes` 后，准确率提升到 `81.00%`
- `mid_14_24_24` 达到了 `87.18%`，但模型大小也增加到 `14608 bytes`

这表明增加卷积层或通道数能够提升模型的表达能力，但代价是内存占用上升。

### 6.2 Accuracy vs MCU Latency

延迟测试结果同样体现了 trade-off。

- `tiny_2` 最快，平均延迟仅 `41006.36 us`
- `mid_12_24` 在保持较好准确率的同时，平均延迟为 `61851.45 us`
- `mid_16_24` 和 `mid_14_28` 的延迟更高，但准确率并未明显优于 `mid_12_24`
- `mid_14_24_24` 的准确率最高，但平均延迟仍控制在 `69963.86 us`，没有成为最慢的模型

这说明更复杂的模型并不一定在所有指标上都更差，关键在于结构设计是否带来了更高的精度收益。

## 7. 结论

本次 4.2 的架构探索表明：

- 小模型具有更低的模型大小和更快的推理速度，但准确率较低
- 中等复杂度模型在准确率、模型大小和延迟之间能够达到更合理的平衡
- 在最终选出的 5 个模型中，`mid_14_24_24` 表现出最好的综合效果，因为它在目标准确率区间内取得了最高的量化准确率，同时保持了可接受的模型大小和 MCU latency

因此，Task 4.2 的核心结论是：

在 TinyML 场景中，模型选择不应只追求最高准确率或最低延迟，而应综合考虑 accuracy、model size 和 on-device latency 三项指标，选择最适合嵌入式部署的架构。
