3. Design Challenge 1: High-Reliability Thermal Monitor and Controller
Build a multi-threaded system on the nRF54L15 DK that monitors temperature using LM335
temperature sensor, detects hardware faults, and manages alarms. Your team are the lead
application architects and therefore the choice of approach and data presentation is your choice,
but your design will be judged on resource efficiency, and stability.

The Requirements: You must implement at least four distinct threads that handle following
responsibilities:

1. Data Acquisition: Sample data at the rate of 100 ms. Detect and handle ADC errors (e.g.
    impossible or out-of-range voltages)
2. Logic Engine: Compute a 1-minute average. Trigger a 50 ms LED blink if the average temp is
    larger than 28°C.
3. Reporting: Output a structured status log to the serial console. For example:

        [TIMESTAMP] Avg: 27.4C | Raw: 27.1C | Mode: NORMAL                  | LED: OFF
        [TIMESTAMP] Avg: 28.2C | Raw: 28.5C | Mode: WARNING                 | LED: BLINKING
        [TIMESTAMP] Avg: --.-C | Raw: 0.05V | Mode: FAULT                   | LED: SOLID


4. Data communication: Periodically transmits a BLE advertisement message that contains:
    •   The 1-minute average and
    •   The system state (NORMAL / WARNING / FAULT). Note that when the system enters state
        “FAULT” it may be impossible to transmit BLE advertisements.

How threads communicate and how they are prioritized should be designed according to your
judgment. The highest-performing systems will demonstrate:

•   Minimal SRAM Footprint: Storing a 60-second history of float data in a large array is
    "expensive." Can you find a way to compute a 1-minute average using only a few bytes?

•   Zero Jitter: The 100 ms sampling must be 100% precise. If any other tasks running on your
    device cause a sampling delay, your priority logic is flawed.

•   Thread Safety: Moving data from the sensor to the brain must be "lossless." If you use global
    variables without protection, you risk data corruption.

•   Deterministic State Transitions: Your system must handle the transition between NORMAL,
    WARNING, and FAULT without ever entering an undefined state.
Students aiming for a higher score should implement the following Extension Tasks (See the
Mark Scheme) to demonstrate advanced mastery of the nRF54L15 and Zephyr RTOS:

1. Live Calibration: Implement a mechanism to update the warning threshold. The choice of
    mechanism is your decision, it could be a command over the serial line, a button press, or a
    command over the network, any combination of the above or a different alternative
    mechanism of your choice. The calibration process must not affect data acquisition from the
    sensor. How do you ensure the Processing thread sees the new threshold safely while it's in
    the middle of a calculation?
2. Drift Detection: Sensor drift occurs when the electrical properties of the sensor change over
    time (due to heat, humidity, or component aging), causing the "zero point" or baseline to move.
    If the 24-hour baseline moves by more than ±2.0𝑜 𝐶 while the device is in a controlled
    environment, flag the DRIFT state. You are encouraged to use memory-efficient algorithms
    like Welford’s Method or a Running Variance to track this shift without storing 24 hours of
    raw data.

What to Include in Your Report: This part should be described in a dedicated section between
3 and 4 pages long. It must prove that your system is stable, efficient, and reliable. Use the
following four sections in your report for this part:

•   RTOS Architecture & Threading: Provide a thread priority table and IPC diagram (e.g.,
    Mutexes, Message Queues) to justify how your design ensures determinism and zero jitter
    for the 100ms sampling rate.
•   Computational Efficiency: Document the specific algorithms used (such as Welford’s
    Method) to achieve low memory usage, proving you minimized the SRAM footprint for long-
    term averaging and drift detection.    You may wish to include and comment on your work’s
    code and memory footprint as reported by the Zephyr build system.
•   Live Calibration & Safety: Detail the implementation of your threshold update mechanism,
    focusing on how you used concurrency control to ensure the system remains stable and
    thread-safe during live adjustments.
•   Systematic Testing: Demonstrate operation through empirical evidence, including timing
    logs for jitter analysis and serial captures showing successful transitions between NORMAL,
    WARNING, and FAULT states.

4. Design Challenge 2: Embedded Gesture Recognition with TinyML
In this challenge, you will work with an existing embedded machine learning system based on the
Arduino Nano 33 BLE Sense and the Magic Wand example provided in the Harvard_TinyMLx package.
A working baseline system is provided and is not assessed. Assessment focuses on your ability to
modify, extend, and reason about an embedded ML system through your own code and experiments.
Please note that you cannot modify the current dataset split ratio.
4.1. Baseline (Not Assessed)
Enclosed is the Jupyter Notebook for the baseline: UoB-IoT-tb2-MagicWand - Colab
please copy it to your own Google Drive.
The provided baseline is the Magic Wand digit recognition example (0–9):
     • Environment: Google colab
     • Sensor: IMU (accelerometer + gyroscope)
     • Task: digit gesture recognition
     • Platform: Arduino Nano 33 BLE Sense
     • Model: CNN-based neural network deployed using TensorFlow Lite Micro
You must successfully run the baseline before starting the assessed tasks, but baseline performance
itself does not contribute to marks.

4.2. Architecture Exploration
The objective of this task is to understand how model the architecture affects accuracy and memory
usage in an embedded ML system. Using the provided training dataset (no personal data at this stage):

    1. Design and train at least 5 different model architectures by modifying:
            o   number of layers,
            o   number of channels per layer,
            o   layer width or kernel size.
    2. For each model, record:
            o   classification accuracy on test dataset on the PC,
            o   the final model size (TensorFlow Lite Quantized).
    3. Plot and analyse the accuracy vs model size trade-off.

What to include in the report

    •   A plot showing accuracy vs model size. The range of model accuracy should cover 70% ~ 90%.
    •   A plot showing accuracy vs model latency on the MCU. Please use the same accuracy range as
        in the sub-task above. A short explanation of the observed trend.

4.3. Personalisation and Stability Analysis
In this task we you are going to study how personal data affects model behaviour, especially prediction
stability on an embedded device. The requirements and the process are outlined below:

    1. Select 2 numbers based on your group ID ( the 1st number is Your_group_id % 10, and the
       2nd one is Your_group_id % 10 + 1 ).
    2. Collect a personal gesture dataset of the two numbers using the given data collection
       procedure (See the Appendix).
    3. Mix your personal data with the original dataset and retrain the model.
    4. Compare the baseline model and your personalised model in terms of:
           o PC-based classification accuracy,
           o on-device prediction stability.
   5. Define and implement a consistency metric, for example repeating the same gesture 10
       times and measuring how consistent the predictions are.

What to include in the report

   •   Accuracy comparison (baseline vs personalised model).
   •   Numerical results of the consistency metric and comparison.
   •   Brief explanation of the results.


5. Extending the System: Adding a New Gesture Class
This task evaluates the scalability of an embedded ML system when new functionality is introduced.

   1. Collect gesture data for one additional class (your assigned letter is given by Your_group_id %
      26 ).
   2. Extend the classifier from 10 classes (digits 0–9) to 11 classes.
   3. Retrain the model and analyse:
          o the impact on digit recognition accuracy,
          o how well the new class is recognised,
          o any confusion between classes.
What to include in the report

   •   Accuracy results for the 11-class model.
   •   Numerical results of the consistency metric of the new letter.
   •   A short discussion on the effect of class extension.

How to collect a personal gesture dataset:

(1)Open up your browser and navigate to: https://tinyml.seas.harvard.edu/magic_wand

(2) If you see a warning that says: Error: This browser doesn't support Web Bluetooth. Try using
Chrome. If you are not using Chrome, we suggest you switch to Chrome. If you are already using
Chrome, then navigate to the following and make sure to “enable” the “experimental web platform
features.” We have found this step to be necessary on older versions of Chrome and users running
the Linux operating system. chrome://flags/#enable-experimental-web-platform-features.




If this problem persists, make sure you are navigating to https://. For some reason, if you do not use
the secure protocol, then the web Bluetooth will not work.

(3) You should then arrive at a webpage that looks something like the following:




(4) You’ll then need to connect your device over Bluetooth. To do that, simply click the blue
Bluetooth button and a pop-up will appear asking to pair. Select your BLE Sense, which should be
called something like “BLESense-XXXX” where XXXX will vary, and click the pair button. Do note that
the course staff has found that sometimes you have to repeat this step twice. Once you are
connected, the Bluetooth button will turn green.




(5) Once your Arduino is paired, it’s time to record some gestures. You’ll notice that every time you
move the Arduino around and then stop, a new gesture is recorded. This is because the gestures are
automatically split up by times when the wand is kept still. These pauses act like spaces between
words, and so when you've finished, a gesture you should stop moving the wand, so that it ends
cleanly. Note that the direction of the gesture matters (e.g., a clockwise circle is different from a
counterclockwise circle)!


These gestures you are drawing will start to show up in the list on the right side of the screen. You
can look at the shapes shown there to understand whether the gestures came out cleanly. A good
rule of thumb is that if you can't tell what the gesture is by looking at it, then a model will have a
hard time recognizing it too.

If you want to delete a recording, simply click the trashcan icon at the top right of each gesture
recording (You may need to delete a lot of spurious recordings that were made as you moved the
Arduino into position between each gesture).

Also, make sure to label all of your gestures for training. You can label each recording by clicking on
the question mark at the top left of each gesture and typing in your label. For example, the
screenshot below shows the label “2” added to the gesture of the number “2” recorded by the
course staff. Make sure you label all of your gestures and note that case matters! Some students
have noted that it can be easier to create one gesture per recording session in order to make sure all
of the gestures are labeled identically but/and the staff has also gotten it to work with multiple
gestures per recording session.

The staff has found that collecting ~20 examples each of 2-3 different gestures often will be enough
data to successfully train a moderately decent magic wand application (i.e., it will work often for
you may not generalize to other users). To help you keep track of how many gestures you recorded,
there is a number in the top right of the screen (e.g., the number 1 as shown in the image below).
Finally, you can upload multiple JSON files to the training script so don’t feel pressured to do all of
your gesture recordings in one shot!
